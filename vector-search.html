Vertex AI — Busca Vetorial e RAG

Esta página é um documento de referência sobre os quatro recursos centrais do Vertex AI (Google Cloud) para construir aplicações de busca semântica e RAG (Retrieval-Augmented Generation). Cada seção explica os campos, valores aceitos, boas práticas e exemplos de código extraídos da documentação oficial.

📚 RAG Engine

Serviço gerenciado que combina retrieval vetorial com um LLM para responder perguntas sobre seus dados. Gerencia corpus, chunking, embedding e geração.

🔍 Vector Search

Motor de busca por similaridade (k-NN aproximado) que consulta um índice vetorial e retorna os vizinhos mais próximos de um vetor de query.

📇 Vector Index

Estrutura que armazena embeddings (vetores) dos seus dados. Configurado com dimensões, algoritmo (Tree-AH ou Brute Force), métrica de distância e tamanho de shard.

🔗 Endpoint do índice

Recurso implantado que expõe um ou mais índices para consultas. Pode ser público (HTTPS) ou privado (VPC Peering). Gerencia escala e disponibilidade.

Fluxo completo: Dados → Embeddings → Vector Index → Deploy no Endpoint → Consulta via Vector Search → RAG Engine usa o resultado como contexto para o LLM.

RAG Engine

O RAG Engine do Vertex AI é um serviço gerenciado de ponta a ponta para Retrieval-Augmented Generation. Ele abstrai toda a pipeline: ingestão de documentos, chunking, geração de embeddings, indexação vetorial, retrieval e geração de resposta com LLM. Você não precisa gerenciar o Vector Index diretamente — o RAG Engine faz isso por você.

Conceitos fundamentais

ConceitoDescrição
RagCorpusColeção de documentos indexados. Equivale a uma "base de conhecimento". Cada corpus tem seu próprio índice vetorial interno.
RagFileDocumento individual dentro de um corpus. Pode ser texto, PDF, HTML, etc. O RAG Engine faz o chunking automaticamente.
RagResourceReferência a um corpus usado em uma query de retrieval ou geração.
RetrievalConfigConfiguração de como o retrieval é feito: quantos chunks retornar, threshold de similaridade, etc.

Fluxo de uso

1. Criar RagCorpus
2. Importar documentos (RagFiles) — GCS, Drive, inline
3. RAG Engine faz chunking + embedding automaticamente
4. Chamar retrieveContexts() com a query do usuário
5. Usar os contextos retornados para chamar generateContent() com o LLM
— OU —
4+5. Chamar diretamente generateContent() com RAG tool configurada

Criar um RagCorpus

from vertexai.preview import rag
import vertexai

vertexai.init(project="SEU_PROJETO", location="us-central1")

# Criar corpus com embedding model padrão
corpus = rag.create_corpus(
    display_name="meu-corpus",
    description="Base de conhecimento de documentos internos",
    # Opcional: especificar modelo de embedding
    embedding_model_config=rag.EmbeddingModelConfig(
        publisher_model="publishers/google/models/text-embedding-004"
    ),
)
print(corpus.name)  # projects/.../ragCorpora/...

Importar documentos

# Importar de Cloud Storage
rag.import_files(
    corpus_name=corpus.name,
    paths=["gs://meu-bucket/documentos/"],
    # Configuração de chunking
    chunk_size=512,        # tokens por chunk (padrão: 1024)
    chunk_overlap=100,     # sobreposição entre chunks (padrão: 200)
)

# Importar de Google Drive
rag.import_files(
    corpus_name=corpus.name,
    paths=["https://drive.google.com/drive/folders/FOLDER_ID"],
    chunk_size=512,
    chunk_overlap=100,
)

Parâmetros de chunking

ParâmetroTipoPadrãoDescrição
chunk_sizeint1024Número de tokens por chunk. Chunks menores = mais precisão; maiores = mais contexto por resultado.
chunk_overlapint200Tokens de sobreposição entre chunks consecutivos. Evita perda de contexto nas bordas.

Retrieval de contextos

response = rag.retrieval_query(
    rag_resources=[
        rag.RagResource(rag_corpus=corpus.name)
    ],
    text="Qual é a política de reembolso?",
    similarity_top_k=5,           # número de chunks a retornar
    vector_distance_threshold=0.5, # threshold de distância (0=idêntico, 1=diferente)
)
for context in response.contexts.contexts:
    print(context.text)
    print(context.score)

Parâmetros de retrieval

ParâmetroTipoDescrição
similarity_top_kintNúmero máximo de chunks retornados. Padrão: 10. Valores maiores aumentam recall mas podem aumentar ruído.
vector_distance_thresholdfloat (0–1)Threshold de distância vetorial. Chunks com distância acima desse valor são descartados. Padrão: 0.5 para DOT_PRODUCT.

Geração com RAG (generateContent)

import vertexai
from vertexai.generative_models import GenerativeModel, Tool
from vertexai.preview.generative_models import grounding

vertexai.init(project="SEU_PROJETO", location="us-central1")

rag_retrieval_tool = Tool.from_retrieval(
    retrieval=grounding.Retrieval(
        source=grounding.VertexRagStore(
            rag_resources=[
                grounding.RagResource(rag_corpus=corpus.name)
            ],
            similarity_top_k=5,
            vector_distance_threshold=0.5,
        )
    )
)

model = GenerativeModel("gemini-2.0-flash-001", tools=[rag_retrieval_tool])
response = model.generate_content("Qual é a política de reembolso?")
print(response.text)

Modelos de embedding suportados

ModeloDimensõesUso recomendado
text-embedding-004768Textos em inglês e multilíngue. Boa relação custo/qualidade.
text-multilingual-embedding-002768Textos em múltiplos idiomas incluindo português.
textembedding-gecko@003768Versão anterior, ainda suportada.
Atenção: O modelo de embedding definido na criação do corpus não pode ser alterado depois. Todos os documentos importados usarão esse modelo. Se precisar mudar, crie um novo corpus.

Corpus: fontes de dados e gestão

O RAG Engine suporta várias fontes para importar documentos no corpus:

FonteUso
Cloud Storage (GCS)paths=["gs://bucket/pasta/"]. PDFs, TXT, HTML. Formato mais comum para pipelines automatizados.
Google Drivepaths=["https://drive.google.com/drive/folders/ID"]. Acesso via OAuth ao Drive do usuário ou conta de serviço.
Conteúdo inlineEnvio de texto ou metadados diretamente na API para indexação sob demanda.

Após a importação, o corpus mantém um índice vetorial interno. Você pode reimportar para adicionar ou atualizar documentos; a gestão do ciclo de vida (ex.: exclusão de arquivos obsoletos) é feita via API ou Console.

Retrieval configs e reranking

RetrievalConfig controla quantos chunks retornar (similarity_top_k), o threshold de distância (vector_distance_threshold) e, quando disponível, opções de reranking para melhorar a relevância dos resultados antes de enviar ao LLM. Consulte a documentação da sua região para parâmetros de reranking disponíveis.

Grounding com Gemini

Grounding é a integração do retrieval com o modelo de linguagem: o Gemini recebe automaticamente os trechos recuperados do corpus como contexto e pode citar as fontes na resposta. Use Tool.from_retrieval() com grounding.Retrieval e VertexRagStore ao chamar generateContent(); o modelo usa o RAG Engine como ferramenta e devolve a resposta já fundamentada nos documentos.

Vector Search

O Vertex AI Vector Search (anteriormente Matching Engine) é o serviço de busca por similaridade vetorial do Google Cloud. Dado um vetor de consulta, retorna os k vizinhos mais próximos de um índice vetorial usando o algoritmo ScaNN (Scalable Nearest Neighbors). Suporta bilhões de vetores com latência de milissegundos.

Métricas de distância (distanceMeasureType)

Definida na criação do índice. Não pode ser alterada depois.

ValorNomeQuando usarObs.
DOT_PRODUCT_DISTANCE Produto Escalar Padrão e recomendado para embeddings de texto normalizados. Equivale ao cosseno quando vetores têm norma L2 = 1. Definido internamente como negativo do produto escalar (menor = mais similar).
COSINE_DISTANCE Distância do Cosseno Quando os vetores não estão normalizados e você quer ignorar a magnitude. Google recomenda usar DOT_PRODUCT_DISTANCE + UNIT_L2_NORM em vez disso — mesmo resultado, mais rápido.
SQUARED_L2_DISTANCE Euclidiana (L2) ao quadrado Dados numéricos em escala similar, clustering, quando a magnitude importa. Sensível a vetores com magnitudes muito diferentes.
L1_DISTANCE Manhattan (L1) Dados esparsos, quando outliers são comuns. Menos comum para embeddings de texto.

Algoritmos de busca

AlgoritmoQuando usarTradeoff
Tree-AH (padrão) Produção, grandes volumes (>1M vetores). Usa árvore de particionamento + Asymmetric Hashing. Alta velocidade, recall configurável via fractionLeafNodesToSearch. Requer approximateNeighborsCount.
Brute Force Desenvolvimento, testes de recall, índices pequenos (<10k vetores). 100% de recall, mas lento para grandes volumes. Sem configuração adicional.

Filtragem de resultados (Restricts)

Permite restringir a busca a subconjuntos do índice usando atributos categóricos (token restricts) ou numéricos (numeric restricts).

Lógica: AND entre namespaces diferentes, OR dentro do mesmo namespace.
Exemplo: {color: red, blue}, {shape: square} = (red OU blue) E (square) 
# Exemplo de query com filtros (Python SDK)
response = my_index_endpoint.find_neighbors(
    deployed_index_id="meu-indice-implantado",
    queries=[[0.1, 0.2, 0.3, ...]],  # vetor de query
    num_neighbors=10,
    filter=[
        # Apenas itens com color=red OU color=blue
        Namespace("color", allow_tokens=["red", "blue"], deny_tokens=[]),
        # Excluir itens com shape=circle
        Namespace("shape", allow_tokens=[], deny_tokens=["circle"]),
    ],
    numeric_filter=[
        # Apenas itens com price < 100
        NumericNamespace(name="price", value_int=100, op="LESS"),
    ],
)

Operadores numéricos disponíveis

OperadorSignificado
LESSValor do datapoint < valor da query
LESS_EQUALValor do datapoint <= valor da query
EQUALValor do datapoint == valor da query
GREATER_EQUALValor do datapoint >= valor da query
GREATERValor do datapoint > valor da query
NOT_EQUALValor do datapoint != valor da query

Crowding (diversidade de resultados)

O crowding tag limita quantos resultados podem vir de um mesmo "grupo". Por exemplo, em um catálogo de produtos, você pode querer no máximo 2 resultados por marca. 

# Limitar a 2 resultados por crowding_tag
response = my_index_endpoint.find_neighbors(
    deployed_index_id="meu-indice",
    queries=[[0.1, 0.2, ...]],
    num_neighbors=10,
    per_crowding_attribute_neighbor_count=2,
)

Busca Híbrida (Dense + Sparse)

Combina busca semântica (dense embeddings) com busca por palavras-chave (sparse embeddings) usando o algoritmo RRF (Reciprocal Rank Fusion). 

from vertexai.matching_engine.matching_engine_index_endpoint import HybridQuery

hybrid_query = HybridQuery(
    dense_embedding=[0.1, 0.2, 0.3, ...],          # embedding semântico
    sparse_embedding_dimensions=[10, 20, 30],        # dimensões do sparse
    sparse_embedding_values=[1.0, 0.5, 0.8],         # valores do sparse
    rrf_ranking_alpha=0.5,  # 0=só sparse, 1=só dense, 0.5=balanceado
)

response = my_index_endpoint.find_neighbors(
    deployed_index_id="meu-indice",
    queries=[hybrid_query],
    num_neighbors=10,
)

Criar um novo índice

No Console do Vertex AI (Vector Search), o formulário Create a new index permite configurar um índice vetorial que será usado para buscas por similaridade. Abaixo, cada campo do formulário é explicado para você preencher com segurança.

Campos principais

Campo Obrigatório Explicação
Display name (Nome de exibição) Sim Nome pelo qual o índice aparece na lista do console. Use um nome descritivo, por exemplo vertex-vector-search-spartan-group1 ou rag-pdf-index. Serve apenas para identificação; o ID interno é gerado automaticamente pelo Vertex AI.
Description (Descrição) Não Texto livre para descrever a finalidade do índice. Ex.: "Índice para RAG do corpus de manuais". Ajuda a documentar o recurso para sua equipe.
Region (Região) Sim Região do Google Cloud onde o índice será criado e processado (ex.: us-east1 (South Carolina), us-central1, southamerica-east1). Deve ser a mesma região do bucket GCS onde estão os dados vetoriais; caso contrário, a criação pode falhar ou ter custos de transferência.
GCS folder URI (URI da pasta GCS) Sim Caminho da pasta no Google Cloud Storage onde estão os dados vetoriais. Formato: gs://BUCKET/PASTA/ (ex.: gs://gcs-spartan-rag). A pasta deve conter arquivos no formato suportado (JSON, CSV ou AVRO) com os embeddings. Use o botão Browse no console para selecionar. Há limite de 5.000 arquivos no diretório raiz.
Algorithm type (Tipo de algoritmo) Sim Algoritmo usado pelo Vector Search para busca eficiente de vizinhos mais próximos.
Tree-AH algorithm: recomendado para produção e grandes volumes; usa árvore de particionamento + Asymmetric Hashing, com recall configurável.
Brute Force: busca exata, ideal para desenvolvimento ou índices pequenos (<10k vetores); não exige parâmetros extras.
Dimensions (Dimensões) Sim Número de dimensões dos vetores de entrada (ex.: 768). Deve ser exatamente igual ao tamanho do embedding gerado pelo seu modelo (ex.: text-embedding-004 usa 768; outros modelos usam 1536, 3072, etc.). Não pode ser alterado após a criação do índice.
Approximate neighbors count (Contagem aproximada de vizinhos) Tree-AH Número padrão de vizinhos que a busca aproximada retorna antes da reordenação exata. Ex.: 200. Deve ser maior que o número de vizinhos que você pede na query (num_neighbors). Regra prática: use algo como 10× o num_neighbors para melhor recall. Valores maiores aumentam recall e latência.
Update method (Método de atualização) Sim Batch: você envia dados via GCS em lotes e o índice é reconstruído conforme um agendamento. Ideal para dados que mudam com pouca frequência.
Stream: permite atualizar e consultar o índice em tempo quase real (inserir, atualizar e deletar vetores via API). Ideal para catálogos dinâmicos ou atualizações frequentes.
Shard size (Tamanho do shard) Não Os dados do índice são divididos em partes iguais chamadas shards. Este campo define o tamanho de cada shard (ex.: Small). Shards maiores usam menos nós e mais memória por nó. Opções típicas: Small (até ~500k vetores de 768 dims), Medium (~5M), Large (~12M). Afeta custo e desempenho; consulte a documentação de preços por tamanho.

Opções avançadas (Advanced options)

Configurações adicionais que influenciam a métrica de distância e o comportamento do algoritmo.

Campo Exemplo Explicação
Distance measure type (Tipo de medida de distância) Dot product distance Métrica usada na busca de vizinhos mais próximos. Valor padrão: Dot product distance. Equivale ao cosseno quando os vetores estão normalizados (L2 = 1). Índices esparsos suportam apenas produto escalar. Outras opções: Cosine, Squared L2, L1 — consulte o painel Vector Search para quando usar cada uma.
Feature norm type (Tipo de normalização) None Tipo de normalização aplicada a cada vetor antes da indexação. Padrão: None. Use Unit L2 norm quando quiser que produto escalar se comporte como similaridade de cosseno (recomendado para embeddings de texto com DOT_PRODUCT_DISTANCE).
Fraction leaf nodes to search (Fração de nós folha para buscar) 0.05 Fração dos nós folha que cada consulta pode percorrer. Deve estar no intervalo (0.0, 1.0). Padrão: 0.05 (5%). Valores maiores tendem a aumentar o recall e a latência; valores menores deixam a busca mais rápida e podem reduzir o recall.
Leaf node embedding count (Contagem de embeddings por nó folha) 1000 Número de embeddings armazenados em cada nó folha da árvore. Padrão: 1000. Mais embeddings por folha resultam em árvore menor, mas mais dados são examinados por folha na busca; menos embeddings por folha deixam a árvore maior e podem melhorar a precisão em alguns cenários.
Resumo: Preencha Display name, Region, GCS folder URI, Algorithm type, Dimensions e Update method como obrigatórios. Para Tree-AH, defina Approximate neighbors count. Use Advanced options para ajustar distância, normalização e parâmetros da árvore.

Vector Index

Um Vector Index é a estrutura de dados que armazena os embeddings (vetores) dos seus itens e permite buscas por similaridade eficientes. É criado a partir de arquivos no Cloud Storage e configurado com parâmetros que definem como a busca será feita.

Tela de criação — campos detalhados

As capturas de tela mostram o formulário "Criar um índice" no Console do Google Cloud. Abaixo, cada campo com seus valores e significados:

Campo Obrigatório Descrição e valores aceitos
Nome de exibição Sim Nome legível para identificar o índice na lista. Pode conter letras, números e hifens. Exemplo: rag-pdf-index. O ID interno é gerado automaticamente.
Descrição Não Texto livre descrevendo o propósito do índice. Exemplo: "Índice para RAG Corpus com Vector Search".
Região Sim Região do Google Cloud onde o índice será criado. Deve ser a mesma região do bucket do GCS com os dados. Exemplo: us-east1, us-central1, southamerica-east1.
gs:// URI da pasta do GCS Sim Caminho do Cloud Storage com os arquivos de dados (embeddings). Formato: gs://BUCKET/PASTA/. A pasta deve conter arquivos .json, .csv ou .avro com os vetores. Limite: 5.000 arquivos no diretório raiz.
Tipo de algoritmo Sim Tree-AH (padrão e recomendado): algoritmo aproximado baseado em árvore + Asymmetric Hashing. Ideal para produção com grandes volumes.
Brute Force: busca linear exata. Ideal para testes e índices pequenos (<10k vetores). Não requer configuração adicional.
Dimensões Sim Número de dimensões dos vetores de entrada. Deve corresponder exatamente ao modelo de embedding usado. Exemplos: 768 (text-embedding-004, BERT), 1536 (OpenAI ada-002), 3072 (text-embedding-3-large). Não pode ser alterado após a criação.
Contagem estimada de vizinhos Tree-AH (approximateNeighborsCount) Número padrão de vizinhos aproximados a recuperar antes do reranking exato. Deve ser maior que o num_neighbors da query. Regra: use 10× o num_neighbors para valores pequenos. Padrão recomendado: 150. Valores maiores = maior recall, maior latência.
Método de atualização Sim Lote (BATCH_UPDATE): atualização em lote via GCS. O índice é reconstruído periodicamente. Ideal para dados que mudam com pouca frequência.
Streaming (STREAM_UPDATE): permite inserir, atualizar e deletar vetores em tempo real via API. Ideal para catálogos dinâmicos.
Tamanho do fragmento Não (ShardSize) Tamanho de cada shard do índice. Shards maiores = menos nós, mais memória por nó.
SHARD_SIZE_SMALL: 2 GiB/shard — até ~500k vetores de 768 dims.
SHARD_SIZE_MEDIUM: 20 GiB/shard — até ~5M vetores de 768 dims.
SHARD_SIZE_LARGE: 50 GiB/shard — até ~12M vetores de 768 dims.

Opções avançadas (TreeAhConfig)

CampoPadrãoDescrição
fractionLeafNodesToSearch 0.05 Fração dos nós folha visitados em cada query (0.0–1.0 exclusivo). Maior = mais recall, mais latência. Exemplo: 0.05 = 5% dos nós folha.
leafNodeEmbeddingCount 1000 Número de embeddings por nó folha. Mais embeddings por folha = árvore menor, mas mais dados examinados por folha.

Normalização de vetores (featureNormType)

ValorDescrição
NONEPadrão. Sem normalização. Use quando os vetores já estão normalizados ou quando usa SQUARED_L2_DISTANCE.
UNIT_L2_NORMNormaliza cada vetor para norma L2 = 1 antes de indexar. Recomendado com DOT_PRODUCT_DISTANCE para equivaler ao cosseno.

Formato dos dados de entrada

Os arquivos no GCS devem seguir um dos formatos abaixo:

Criar índice via Python SDK

from google.cloud import aiplatform

aiplatform.init(project="SEU_PROJETO", location="us-central1")

# Criar índice Tree-AH com DOT_PRODUCT + normalização L2
my_index = aiplatform.MatchingEngineIndex.create_tree_ah_index(
    display_name="meu-indice-producao",
    contents_delta_uri="gs://meu-bucket/embeddings/",
    dimensions=768,
    approximate_neighbors_count=150,
    distance_measure_type="DOT_PRODUCT_DISTANCE",
    feature_norm_type="UNIT_L2_NORM",
    leaf_node_embedding_count=1000,
    leaf_nodes_to_search_percent=5,
    description="Índice de produção para busca semântica",
    shard_size="SHARD_SIZE_MEDIUM",
)
print(my_index.resource_name)

Update e rebuild de índice (Batch)

Em modo Lote (BATCH_UPDATE), o índice é atualizado enviando novos arquivos ao GCS e chamando update_embeddings(). Você pode fazer atualização incremental (is_complete_overwrite=False) ou substituição total (True). O rebuild pode levar minutos ou horas conforme o tamanho dos dados; o índice permanece disponível durante o processo em muitas configurações. 

# Atualização incremental (adiciona/atualiza vetores)
my_index.update_embeddings(
    contents_delta_uri="gs://meu-bucket/atualizacoes/",
    is_complete_overwrite=False,  # False = incremental, True = substituição total
)

# Substituição completa do índice
my_index.update_embeddings(
    contents_delta_uri="gs://meu-bucket/novo-corpus-completo/",
    is_complete_overwrite=True,
)

Atualização em Streaming (tempo real)

# Upsert (inserir ou atualizar) vetores individuais
my_index.upsert_datapoints(
    datapoints=[
        aiplatform.matching_engine.MatchingEngineIndex.Datapoint(
            datapoint_id="doc-novo-001",
            feature_vector=[0.1, 0.2, 0.3, ...],
            restricts=[
                aiplatform.matching_engine.MatchingEngineIndex.Datapoint.Restriction(
                    namespace="categoria",
                    allow_list=["tecnologia"],
                )
            ],
        )
    ]
)

# Remover vetores
my_index.remove_datapoints(datapoint_ids=["doc-001", "doc-002"])
Importante: O índice precisa estar implantado em um endpoint ativo para que as atualizações de streaming sejam refletidas imediatamente nas queries. Para índices em lote, as atualizações só ficam disponíveis após a reconstrução do índice.

Criar um endpoint de índice

No console do Vertex AI (Vector Search), o formulário Criar um endpoint de índice permite provisionar o recurso que receberá o deploy dos seus índices vetoriais. Abaixo, cada campo e cada opção de acesso são explicados. Importante: a forma de acesso ao endpoint não pode ser alterada após a criação — escolha com cuidado.

Campos do formulário

Campo / Seção Obrigatório Explicação
Nome de exibição Sim Nome pelo qual o endpoint aparecerá na lista do console. Use um nome descritivo (ex.: vector-search-endpoint-prod). O asterisco indica preenchimento obrigatório. Esse nome ajuda a identificar o recurso quando houver vários endpoints no projeto.
Local — Região Sim Região do Google Cloud onde o endpoint será provisionado. Ex.: us-east1 (Carolina do Sul), us-central1, southamerica-east1. O endpoint e os índices implantados nele devem estar na mesma região para consultas funcionarem. Escolha uma região próxima aos seus usuários ou ao restante da sua infraestrutura.
Seção Acesse — determina como o endpoint pode ser acessado. Não será possível alterar após a criação.
Padrão Opção Disponibiliza o endpoint usando uma API REST. O endpoint fica acessível pela internet via HTTPS (API REST pública). Mais simples de configurar e ideal para protótipos, desenvolvimento ou quando não há requisitos de rede privada. As chamadas são autenticadas (ex.: Google Cloud IAM, API Key).
Private Service Connect (recomendado) Opção Cria uma conexão particular com o endpoint usando Private Service Connect em uma rede VPC. O tráfego não sai pela internet pública; a comunicação ocorre dentro da sua VPC. Recomendado para produção e ambientes corporativos, onde segurança e isolamento de rede são importantes. Exige configuração de Private Service Connect e de um endpoint de serviço na VPC.
Acesso a serviços particulares Opção Cria uma conexão particular com o endpoint usando uma rede VPC e o acesso a serviços particulares. Outro modo de acesso privado, baseado no recurso de “Private Google Access” / serviços particulares na VPC. Use quando sua política de rede exigir esse modelo em vez do Private Service Connect. Consulte a documentação para requisitos de DNS e de rede.
Atenção: A forma de acesso (público REST, Private Service Connect ou acesso a serviços particulares) não pode ser alterada depois que o endpoint for criado. Se precisar mudar, será necessário criar um novo endpoint e implantar os índices novamente.
Resumo: Preencha o Nome de exibição e escolha a Região. Em Acesse, escolha Padrão (API REST pública), Private Service Connect (recomendado para produção) ou Acesso a serviços particulares. Em seguida, use Criar para provisionar o endpoint; depois, implante os índices na seção "Endpoint do índice".

Endpoint do índice

Um Index Endpoint é o recurso que expõe um ou mais índices vetoriais para consultas. Para usar o Vector Search, você precisa: (1) criar um endpoint, (2) implantar (deploy) um índice nele, (3) enviar queries para o endpoint.

Tipos de endpoint

TipoAcessoQuando usar
Endpoint público HTTPS público com autenticação IAM. Domínio: NÚMERO.REGIÃO-PROJETO.vdb.vertexai.goog Aplicações que precisam acessar de fora da VPC, Cloud Run, Cloud Functions, ambientes externos.
Endpoint privado (VPC) Acessível apenas dentro da VPC via VPC Peering. IP interno. Produção com requisitos de segurança, menor latência, GKE, Compute Engine na mesma VPC.

Tela "Implantar índice" — campos detalhados

A terceira captura mostra o diálogo de implantação. Campos:

CampoObrigatórioDescrição
Nome de exibição Sim Nome do índice implantado (DeployedIndex). O ID é gerado com base nesse nome e não pode ser alterado depois. Use nomes descritivos como rag-pdf-v1.
Endpoint Sim Endpoint onde o índice será implantado. Pode ser um existente ou criar um novo. Um endpoint pode ter múltiplos índices implantados simultaneamente.
Tipo de máquina Não Tipo de VM para servir o índice. Opções disponíveis:
Padrão:
 • n1-standard-16: 16 vCPUs, 60 GiB RAM — uso geral
 • n1-standard-32: 32 vCPUs, 120 GiB RAM — alto throughput
 • e2-standard-16: 16 vCPUs, 64 GiB RAM — custo-benefício
 • n2d-standard-32: 32 vCPUs, 128 GiB RAM — AMD, bom custo
Memória elevada: para índices muito grandes que precisam de mais RAM.
Se não especificado, o padrão é n1-standard-16.
Réplicas mínimas Não Número mínimo de réplicas (nós) sempre ativas. Padrão: 2. Mais réplicas = maior disponibilidade e throughput, mas maior custo.
Réplicas máximas Não Número máximo de réplicas para autoscaling. O sistema escala automaticamente entre mínimo e máximo conforme a carga.

Criar endpoint e implantar índice (Python)

from google.cloud import aiplatform

aiplatform.init(project="SEU_PROJETO", location="us-central1")

# 1. Criar endpoint público
my_index_endpoint = aiplatform.MatchingEngineIndexEndpoint.create(
    display_name="meu-endpoint-producao",
    description="Endpoint público para busca semântica",
    public_endpoint_enabled=True,  # True = público, False = privado VPC
)

# 2. Implantar índice no endpoint
my_index_endpoint.deploy_index(
    index=my_index,
    deployed_index_id="rag-pdf-v1",  # ID único, imutável
    display_name="RAG PDF v1",
    machine_type="n1-standard-16",
    min_replica_count=2,
    max_replica_count=10,
)

print(my_index_endpoint.public_endpoint_domain_name)
# Saída: 1234567890.us-central1-123456789.vdb.vertexai.goog

Parâmetros de performance em tempo de query

ParâmetroAPIImpacto
num_neighbors / neighbor_count Por query Número de resultados retornados. Valores até 300 são performáticos na maioria dos casos. Para valores maiores, teste com sua carga.
approximateNeighborsCount / approximate_neighbor_count Por query (override) Quantos vizinhos aproximados recuperar antes do reranking. Deve ser > num_neighbors. Regra: 10× para valores pequenos. Aumentar = mais recall, mais latência.
fractionLeafNodesToSearch / fraction_leaf_nodes_to_search_override Por query (override) Fração dos nós folha visitados (0.0–1.0). Override do valor configurado no índice. Aumentar = mais recall, mais latência e custo.
per_crowding_attribute_neighbor_count Por query Máximo de resultados com o mesmo crowding_tag. Aumenta diversidade dos resultados.

Criar um bucket (Cloud Storage)

No Google Cloud Console, a tela Criar um bucket do Cloud Storage é dividida em várias etapas. Abaixo, cada seção e cada opção são explicadas para você configurar o bucket com segurança — inclusive para uso com Vector Search (pasta GCS onde ficam os dados vetoriais do índice).

1. Comece já — Nome e marcadores

ItemExplicação
Nome do bucket Nome globalmente único no Google Cloud. Exemplos: exemplo, exemplo_intervalo-1, exemplo.com. Siga as Diretrizes de nomenclatura (somente letras minúsculas, números, hífens e underscores; 3–63 caracteres). Dica: não inclua informações confidenciais no nome — ele aparece em URLs e logs.
Marcadores (opcional) Labels (chave-valor) para organizar e filtrar buckets na lista e em relatórios (ex.: ambiente=producao, projeto=rag). Útil para custos e políticas da organização.

2. Escolha onde armazenar seus dados

Define a localização dos dados; afeta custo, desempenho e disponibilidade. Não pode ser alterada após a criação.

ItemExplicação
Tipo de local — Multi-region Maior disponibilidade em uma área ampla (ex.: US, EU). Ideal para dados que precisam de redundância geográfica.
Tipo de local — Dual-region Alta disponibilidade e baixa latência em duas regiões. Bom equilíbrio entre custo e resiliência.
Tipo de local — Region Latência mais baixa em uma única região (ex.: us-east1 (Carolina do Sul)). Recomendado para Vector Search: use a mesma região do índice e do endpoint.
Região (dropdown) Se você escolheu Region ou Dual-region, selecione a região (ou par). Para RAG/Vector Search, alinhe com us-central1, us-east1, southamerica-east1, etc.
Replicação entre buckets (Transfer Service) Checkbox opcional: replica dados para outro bucket conforme são adicionados ou alterados. Permite ter o mesmo conteúdo em outro bucket com classe ou região diferente.

3. Escolha como armazenar seus dados

A classe de armazenamento impacta custo, recuperação e operações. Escolha entre gerenciamento automático ou uma classe padrão fixa.

ItemExplicação
Autoclass Transfere automaticamente cada objeto para Standard ou Nearline conforme a atividade. Recomendado quando a frequência de uso é imprevisível. Pode ser alterado para classe padrão depois.
Definir uma classe padrão Aplica a classe escolhida a todos os objetos do bucket (a menos que você mude por objeto ou use regras de ciclo de vida). Recomendado quando o uso é previsível.
Standard Melhor para dados acessados com frequência e armazenamento de curto prazo. Custo maior por GB, sem taxa de recuperação. Ideal para dados de ingestão e índices Vector Search em uso ativo.
Nearline Para backups e dados acessados menos de uma vez por mês. Custo de armazenamento menor, com taxa de acesso.
Coldline Recuperação de desastres e dados acessados menos de uma vez por trimestre. Armazenamento mais barato, maior custo de recuperação.
Archive Preservação digital a longo prazo; dados acessados menos de uma vez por ano. Menor custo de armazenamento, maior custo e latência de recuperação.

4. Escolha como controlar o acesso a objetos

ItemExplicação
Impedir acesso público Aplicar a prevenção do acesso público neste bucket: impede que os dados sejam acessados publicamente pela internet e que o bucket seja usado para hospedagem web. Recomendado para dados internos e para buckets de RAG/Vector Search (os dados são acessados apenas por serviços autorizados, ex.: Vertex AI).
Controle de acesso — Uniforme Permissões definidas apenas no nível do bucket via IAM; aplicadas uniformemente a todos os objetos. Mais simples e alinhado às boas práticas. Importante: após ativado, torna-se permanente após 90 dias. Muitas organizações exigem essa opção.
Controle de acesso — Detalhado Permite ACLs (listas de controle de acesso) por objeto, além do IAM no bucket. Controle mais granular, porém mais complexo. Use apenas se precisar de permissões diferentes por objeto.

5. Escolha como proteger os dados do objeto

Camadas extras de segurança e recuperação além da proteção padrão do Cloud Storage.

ItemExplicação
Política de exclusão reversível Permite recuperar objetos e o bucket após uma exclusão. Objetos excluídos são retidos por um período. Duração padrão: 7 dias (ou o valor definido pela organização). Personalizada: defina outro período; duração 0 desativa a exclusão reversível (exclusões permanentes).
Controle de versões do objeto Mantém várias versões de um objeto; permite restaurar versões substituídas ou excluídas. Para custos, limite o número de versões não atuais e use regras de ciclo de vida para expirar versões antigas.
Retenção Impede exclusão ou modificação de objetos por um período predefinido. Usado para conformidade regulatória (compliance).
Criptografia de dados Seção opcional para configurar chaves de criptografia. Por padrão, o GCS usa criptografia gerenciada pelo Google; é possível usar chaves próprias (CSEK/CMEK) para requisitos de compliance.
Para Vector Search / RAG: crie o bucket na mesma região do índice e do endpoint; use classe Standard para dados em uso ativo; ative impedir acesso público e prefira acesso uniforme. A pasta GCS (gs://bucket/pasta/) será informada na criação do Vector Index.
